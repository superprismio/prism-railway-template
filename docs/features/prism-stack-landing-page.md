# Prism Stack Landing Page

Working draft for the Prism Stack marketing page and related launch copy.

## Positioning

Prism Stack is infrastructure for collaborative AI in communities.

It is not a single-purpose chatbot. It is a deployable stack for teams that need:

- shared context across chat, meetings, docs, and repos
- AI assistance that works at the community level, not just the individual level
- control over where data lives and how agents interact with it
- a practical path from Discord-first workflows to broader community infrastructure

The current entry point is Discord, but the product direction is broader:

- Discord for chat, meetings, and operator workflows
- Prism Memory for shared memory and knowledge
- Codex Runtime for operator and agent runs
- a stack that can be adapted to other transports, tools, and environments

## Voice

The page should feel:

- research-informed, but not academic
- systems-oriented, but not abstract
- ambitious, but concrete
- collaborative, not individual-productivity focused

The Superprism site frames the core problem well:

- most AI tools are built for individuals

- context, privacy, and coordination matter more than isolated chat UX

Prism Stack should keep that posture, but shift from R&D lab framing to deployable stack framing.

## Core Message

Prism Stack helps communities build shared intelligence.

Communities already produce context in Discord channels, meeting transcripts, docs, repos, and ongoing contributor work. That context is usually fragmented, hard to search, hard to carry forward, and hard for AI systems to use without becoming noisy or invasive.

Prism Stack gives communities a way to:

- ingest and organize shared context
- run memory and knowledge workflows over that context
- let operators and contributors interact with agents inside existing workflows
- preserve more sovereignty over data, tooling, and deployment shape

## Problem Statement

### The problem with community coordination today

Communities do their real work across:

- Discord chats and threads
- voice meetings
- docs and handbooks
- GitHub repos
- contributor memory

The result is predictable:

- important decisions disappear into chat
- meeting context gets lost after the call
- newcomers cannot reconstruct why things work the way they do
- operators repeat the same explanations and synthesis work
- AI tools help individuals, but rarely help the group maintain shared context

### The problem with most AI tools

Most AI products assume:

- one user
- one chat window
- one private context

That does not match communities.

Communities need:

- shared memory
- durable context
- scoped knowledge
- human review and control
- workflows that fit into real coordination environments

## What Prism Stack Solves

### 1. Shared context does not stay trapped in chat

Prism Stack can ingest community activity and turn it into:

- memory inbox items
- rolling memory
- searchable knowledge
- meeting summaries
- reusable artifacts

This reduces repeated context loss.

### 2. Meetings become durable and usable

Discord voice sessions can produce:

- transcripts
- summaries
- artifacts linked back into the stack

This makes meetings part of the community knowledge surface instead of dead air after the call ends.

### 3. AI can operate on community context, not just private prompts

Agents can work with:

- shared memory
- synced knowledge sources
- community-specific docs
- operator instructions

This enables community-level assistance instead of isolated personal usage.

### 4. Communities keep more control over their own infrastructure

Prism Stack is deployable infrastructure, not just a hosted assistant.

That matters for communities that care about:

- privacy
- internal knowledge boundaries
- custom workflows
- long-term ownership of context

### 5. Operator workflows can happen inside the same system

The stack supports:

- admin workflows
- configuration through chat
- memory operations
- change-request and runtime workflows

This reduces the split between "AI feature" and "operations overhead."

## Key Highlights

### Discord-first, but not Discord-only

Start where communities already work. Build toward a broader context layer later.

### Shared memory plus knowledge

Not just chat history. Not just document search. Both.

### Meeting capture and synthesis

Voice conversations can become searchable artifacts and summaries.

### Configurable ingestion and policy

Communities can shape what gets emphasized, what gets deprioritized, and how memory workflows behave.

### Deployable and adaptable

Useful as a Railway template today. Flexible enough to evolve into other deployment models and other transports.

## Use Cases

### Open source communities

- preserve technical and governance context
- reduce repeated onboarding work
- turn chat and docs into usable shared knowledge

### Research collectives

- maintain continuity across experiments, discussions, and notes
- support synthesis without centralizing everything in one operator
- keep context portable

### DAOs and distributed teams

- capture governance and contributor context
- summarize meetings and active work
- support coordination across asynchronous channels

### Agencies and studios

- maintain client, project, and team context
- reduce context loss between calls and project threads
- support reusable operational memory

### Local-first or sovereignty-conscious organizations

- retain more control over infrastructure and data boundaries
- avoid outsourcing the entire coordination layer to third-party SaaS
